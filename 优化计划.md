# PML 代码优化计划

基于全面代码审阅（45 项发现）整理。

---

## 阶段划分

```
Phase 1 (安全清理) ──→ Phase 2 (架构修整) ──→ Phase 3 (性能/质量)
       │                      │                        │
       ├ 重复代码消除          ├ 循环依赖解除            ├ UV 求解质量
       ├ 未防护的 double_val   ├ 注册表模式              ├ 尾调用优化
       ├ 残留 debug 输出       ├ Value 类型安全          ├ 顶点去重 O(N²)
       └ 裸指针/裸 new        ├ 纹理管线拆分            └ 曲线自适应细分
                               ├ TextureCache 重构
                               └ 全局变量消除
```

各 phase 内部可并行实施。Phase 1 无架构变更风险，可随时合并。Phase 2 涉及接口变更，需确保向后兼容。

## Phase 1: 安全清理（零架构变更）

### 1.1 消除重复代码

| ID | 文件 | 问题 | 方案 |
|----|------|------|------|
| C001 | `evaluator.cpp` + `special_forms.cpp` | `deep_equal` 函数两份完全相同的实现（匿名 namespace） | 提取到 `builtins_helpers.h`，两处删除重复 |

### 1.2 修复未防护的 `double_val()` 调用

审阅发现大量 `double_val()` 调用不先检查 `is_int()`，当 PML 字面量是整数时会得到错误结果（已在 `triangulation.cpp` 触发 bug）。

| ID | 文件 | 行号 | 危险程度 |
|----|------|------|----------|
| C002 | `color_builtins.cpp` | 72 | 高：`args[3].double_val()` 无防护 |
| C003 | `kwargs.h` | 101 | 高：`it->second.double_val()` 无防护 |
| C004 | `skia_backend_internal.h` | 186 | 中：`elems[i].double_val()` |
| C005 | `shader_builtins.cpp` | 多处 | 中：多个无防护 `double_val()` |
| C006 | `api.cpp` | 111 | 中：`return val.double_val()` |

**方案**：给 `Value` 新增 `to_double()` 方法，内部统一做 `is_int()` / `is_double()` 分派，然后全局替换。

### 1.3 残留 debug 输出

| ID | 文件 | 行号 | 问题 |
|----|------|------|------|
| C007 | `render.cpp` | 422, 429 | `fprintf(stderr, ...)` 在生产路径中，动画渲染时输出 |
| C008 | `textured_draw.cpp` | 多处 | 已移除，但需确认 |

**方案**：删除或用 `#ifndef NDEBUG` 包裹。

### 1.4 裸 new/delete

| ID | 文件 | 问题 | 方案 |
|----|------|------|------|
| C009 | `triangulation.cpp:192-199` | poly2tri 点用 `new` 分配，`delete` 手动清理，中间异常不安全 | 用 `unique_ptr` + 自定义 deleter 或 vector 管理 |

### 1.5 重复 include

| ID | 文件 | 行号 | 问题 |
|----|------|------|------|
| C010 | `skia_backend_draw.cpp` | 19, 29 | `textured_draw.h` 被包含两次 |

## Phase 2: 架构修整

### 2.1 解除循环依赖

#### P001 — `pml_core` 依赖 Skia headers

**现状**：`texture_cache.cpp` 在 `pml_core` 中编译，使用了 `sk_sp<SkImage>`，所以 `core/CMakeLists.txt` 必须 `target_link_libraries(pml_core PUBLIC skia_headers)`。这导致整个核心基础库都知道 Skia 类型。

**方案 A（推荐）**：将 `texture_cache` 移到 `pml_backend` 层，`pml_core` 只保留纯接口或前向声明。

**方案 B（轻量）**：用 `void*` opaque pointer 包装 `SkImage`，`pml_core` 只操作 opaque handle。

#### P002 — `pml_graphics` 反向依赖 `pml_evaluator`

**现状**：`render.cpp` 包含 `environment.h` 并调用 `env->define()`，因为 `register_render` 函数在 graphics 层但需要访问 evaluator 的环境。通过 `target_include_directories(pml_graphics PRIVATE ../evaluator)` 实现，这是对单向依赖规则的破坏。

**方案**：将 `register_render` 函数移到 `pml_api` 层，该层天然链接所有模块。

#### P003 — `merge_skin_bindings` 前向声明逃避循环依赖

**现状**：`render.cpp` 内部前向声明了 skeleton 层的函数，注释写明"避免创建目标依赖循环"。

**方案**：在 `pml_skeleton` 中提供独立接口头文件，或通过回调注册。

### 2.2 `draw_object` 注册表模式

**现状**：`skia_backend_draw.cpp` 的 `draw_object()` 有一个 20+ 分支的 if/else 链：

```
if (ParamKey::uv)    → draw_textured_object
if ("group")         → draw_group
if ("rough_*")       → draw_rough_X
if ("rect")          → draw_rect
if ("circle")        → draw_circle
...
```

**方案**：注册表 `std::unordered_map<std::string, DrawFn>`。形状注册自己的绘制函数，`draw_object` 查表分派。新形状无需修改核心文件。

**文件**：`skia_backend_draw.cpp`（主要在 `draw_object` 函数，约 1260-1400 行）

### 2.3 `Value` 类型安全

**现状**：
- `Value` 24+ 个构造函数重载，每个 Box::Kind 一个
- `int_val()` / `double_val()` 无 tag 断言，UB 风险
- `is_int()` / `is_double()` / `as_int()` / `as_double()` 方法散落各处

**方案**：
1. 给 `int_val()` / `double_val()` 加 `assert(tag_ == Tag::Int/Double)`
2. 添加 `to_double()` 统一分派方法
3. 考虑用模板构造函数减少重载

### 2.4 纹理管线拆分

**现状**：`draw_textured_object()` 一个函数内完成四件事：

```
triangulate_shape() → solve_XX_uv() → bake_texture() → drawVertices()
```

中间结果不可单独测试，错误定位困难。

**方案**：拆分为独立阶段函数，每个返回可断言的结构体：

```cpp
Result<TriangulatedMesh>    stage_triangulate(const GraphicObject&);
Result<UVResult>           stage_solve_uv(const TriangulatedMesh&, int mode);
std::shared_ptr<SkImage>   stage_bake(const TextureBox&);
Result<void>               stage_draw(SkCanvas*, const TriangulatedMesh&,
                                      const UVResult&, const SkImage*);
```

每个阶段可加 GTest。

### 2.5 `TextureCache` 重构

**现状**：`TextureCache::instance()` 是 Meyer's 单例，与 `PMLContext` 线程本地的设计不一致。

**方案**：将 `TextureCache` 实例挂到 `PMLContext` 上，使其 per-context 生命周期管理。

### 2.6 全局变量消除

| ID | 变量 | 文件 | 问题 | 方案 |
|----|------|------|------|------|
| C011 | `g_macro_depth` | `evaluator.h:34` | 非 `thread_local`，多线程竞态 | 改为 `thread_local` 或移到 Environment 成员 |

### 2.7 `let` 语义调整

**现状**：`let` = 并行绑定，`let*` = 串行。这与大多数 Lisp（Scheme 的 `let` 是并行的，但用户的直觉期望 `let` 是串行的）。

**方案**（已在计划中）：交换 `let` 和 `let*` 语义，新增 `let-par` 提供并行绑定。

## Phase 3: 性能与质量

### 3.1 UV 求解质量

| ID | 问题 | 严重度 | 方案 |
|----|------|--------|------|
| C012 | `uv_solver.cpp:225` uniform 权重 → UV 变形 | medium | 实现 cotangent Laplacian 权重 |
| C013 | `uv_solver.cpp:115-122` 边界顺序不可靠 | medium | 按角度或相邻关系排序边界顶点 |
| C014 | 默认 harmonic UV 在动态图形上性能差 | medium | 简单形状自动回退到 planar UV；或缓存 UV 结果 |

### 3.2 三角化改进

| ID | 问题 | 严重度 | 方案 |
|----|------|--------|------|
| C015 | `triangulation.cpp:180` 圆/椭圆 N=32 硬编码 | low | 根据半径/曲率动态选择分段数 |
| C016 | `triangulation.cpp:278-283` 顶点去重 O(N²) | medium | 用坐标哈希的空间索引 |
| C017 | 曲线细分 N=8 固定 | low | 基于曲线长度自适应细分 |

### 3.3 尾调用优化

**现状**：`trampoline` 实现了 TCO 循环，但只对 `apply_function` 返回 `TailCall` 时工作。`eval_if`、`eval_cond`、`eval_when`、`eval_let*` 等在尾位置仍会增长 C++ 栈。

**方案**：将这些特殊形式的尾位置 `evaluate(expr, env)` 改为返回 `TailCall{expr, env}`。

### 3.4 非 Windows 字体

**现状**：`skia_backend_draw.cpp:404` 非 Windows 上使用 `SkFontMgr::RefEmpty()` 然后尝试 Arial/Microsoft YaHei → 几乎肯定失败。

**方案**：改用 `SkFontMgr::RefDefault()`，尝试 "sans-serif"、"DejaVu Sans" 等通用字体。

## 验证方式

1. **增量构建**：每个变更后 `cmake --build --preset debug` 通过
2. **回归测试**：`ctest --preset debug` 全部通过（314 tests + 284 smoke）
3. **纹理示例**：`examples/13-texture/*.pml` 全部正确渲染
4. **内存检查**：Phase 1.4 后检查 AddressSanitizer 无泄漏
5. **架构验证**：Phase 2 后检查 `include what you use` 无反向依赖
